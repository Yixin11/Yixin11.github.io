计算机进阶-后台基础

Posted on 2020-11-13 Edited on 2020-11-17 In 后台开发
（大致完成）计算机基础-后台开发。简单介绍一些后台开发中经常涉及的概念。

海量服务

现代互联网已经进入了海量服务时代。海量用户带来的海量数据对后台服务带来前所未有的挑战。

为海量用户提供及时、稳定、可靠的后台服务，是当今后台开发工程师的共同使命与愿景。

高并发

缓存

缓存可以说是高并发读场景下的银弹。对于用户而言，缓存可以节省等待时间，节省数据流量；对于后台服务而言，合理利用缓存能够减少后台服务的压力，加快请求的处理时间，削减服务器成本，减少网络传输的消耗。

常见的缓存有：浏览器缓存，CDN，网关缓存，负载均衡缓存、逻辑层缓存、存储层缓存等。

HTTP 头部中的缓存：
HTTP 1.0 if-modified
HTTP 1.1 etag
HTTP return code 304

后台服务中，如果没有及时更新缓存，就会出现缓存与数据库不一致的情况，此时的缓存的数据便成了脏数据。业界上通用的做法是，先删除缓存，再更新数据库，再删除缓存（延时双删）。

穿透、击穿与雪崩
(1) 如果缓存没有命中，去数据库查则称为穿透，为了避免重复查询不存在数据造成额外的消耗，可以将不存在数据也加入缓存。
(2) 如果 hot key 过期，造成大量数据查询落到数据库中，称为击穿；
(3) 如果同一时刻大量缓存失效，导致大量查询落到数据库中，由此导致查询时间延长，服务阻塞并且请求总处理时间增长，最后服务超时。这种情况称为雪崩。

高并发写

尽量减少写过程中锁的争用。为了保证数据的一致性，数据库的写操作一般要加锁。
一般采取分库分表分区，将锁的消耗尽量减少。也可以采用读写分离。
如果写请求比较少，可以将操作放进消息队列中有序执行，如果短时间写请求较多，放弃数据库的强一致性而保证最终一致性。
如果确实消耗比较大，放弃数据库的强一致性而保证最终一致性。

如果长时间写请求较多又没法处理，可以选择招聘一个架构师。

限流算法

  • 令牌桶
    建立一个带有容量的桶，等速率注入令牌，每次请求在处理前都向桶中尝试获取令牌。如果获得令牌则处理，没获得则直接丢弃。
    因为令牌桶的容量是有限的，如果桶已经满了，则停止注入令牌。

  • 滴桶
    把请求当作水流，所有的请求都被注入一个桶中。桶会以等速率转发桶中的请求。也就是桶中的请求都会被处理。
    如果注入的请求超过了桶的容量，多余的请求都会被丢弃。

线程池、连接池

  • 池子的设计思想：一个对象的生命周期是创建-应用-销毁。但是创建和销毁会消耗大量的资源。
    因为对象是由结构体实例化，所以预先开辟一块内存用来存储对象。销毁对象时将对象放入协程池，需要时再从池中直接获得一个对象。
    池子技术一般用于频繁的创建与销毁对象的后台服务中。

一般不用协程池，因为协程的创建和销毁代价极小。在 golang 中，有使用大小为 32 的协程列表用于协程的调度。

限流、降级、熔断与削峰

  • 发送方统计对方服务的延时率，如果超时率增加则主动降低 qps，称为限流；
  • 接收方统计服务整体延时率，如果超时率超过预设值则主动降级。降级一般先从外围开始降，保证核心服务的正常运行；
  • 服务下游出现严重超时则主动熔断，保证其他模块不会受到影响；
  • 某一时间请求出现峰值，则变并行为串行，延长时间进行处理。一般使用消息队列。

串行无锁：消息队列

消息队列一般使用 kafka、RabbitMQ，变并行为串行；
结构无锁主要是 CAS

微服务

传统业务架构难以服务海量用户。
传统业务架构具有开发简单、快速、成本低的优点，但是应对海量服务却有些招架不住。一方面，传统架构不能及时适应业务的快速增长，不管是横向扩容或者是纵向扩容，都会伴随配置文件的大量更改。伴随时间的增长，业务的迭代和维护就成为一件颇为麻烦的事。另一方面，如果没有良好的设计，后台模块间很容易相互依赖耦合，一旦出现单点故障，整个后台服务都会出现影响。

面对海量服务的业务场景，微服务作为一种设计思想，在后台领域广泛传播，并取得了一定的认可。
但首先需要澄清的是，微服务并不是云服务，也不是任何一种框架。
微服务是一种设计思想，遵循“高内聚，低耦合”的思想，以单一责任原则将传统服务划分为若干模块，佐以注册中心完成服务的注册与发现，模块通过注册中心统一配置和调用，以达到业务拆分的目的。

微服务实现业务拆分来实现业务解耦，同时通过注册中心减少模块间的相互依赖。注册中心的功能是提供服务的自动注册与自动剔除，通过映射与负载均衡进行服务发现和使用心跳包进行容器保活。因为注册中心面向的是微服务的全体模块，所以注册中心必须保证高可用，在业务流量较高的情况下还需要使用多节点保证一致性。

常用的服务注册与发现有 DNS、zookeeper、consul 和 etcd。如果不使用注册中心完成服务发现，则需要将对应模块写入配置文件中，通过直接访问 ip 完成服务发现。

使用微服务将业务拆分有诸多好处。使用 Namespace 调用模块可以隐藏模块的实现细节，服务的扩缩容可以通过服务发现和负载均衡控制；“高内聚，低耦合”的模块有利于开发人员的职责细分，提高协同工作的效率，也降低了修改模块的成本；单一服务出现问题大多不会影响全局服务正常运行。

微服务不是软件设计的银弹。
如果微服务的模块功能过于细分，也会对服务的迭代和维护造成颇为麻烦的局面。因为每次的服务注册与发现大多通过注册中心调度，频繁的调用也会给请求链路带来额外的消耗。因为服务涉及的模块和机器较多，追踪问题也变得更加棘手。
因为微服务涉及的模块较多，对于自动化运维也有一定的要求。维护过多的模块会给运维带来额外的工作。
